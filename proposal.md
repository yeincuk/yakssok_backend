# 약쏙(Yakssok) 의료 데이터 보안 강화 및 아키텍처 개선 제안서 (proposal.md)

의료 및 건강 관련 데이터를 다루는 어플리케이션은 사용자의 민감한 개인정보(PII)와 건강정보(PHI)를 보호하기 위해 금융권 수준의 높은 보안이 요구됩니다. 현재 시스템의 아키텍처를 점검하고, 안전하고 확장 가능한 의료 서비스로 거듭나기 위한 보안 및 구조 강화 방안을 제안합니다.

## 1. 처방전 업로드 시 중간 탈취(MITM) 예방 방안
사용자가 처방전을 촬영하고 서버로 전송하는 과정에서 해커가 악의적인 Wi-Fi나 프록시를 통해 네트워크 패킷을 가로채는 것을 막아야 합니다.

* **완전한 HTTPS 및 HSTS 적용**: 클라이언트(앱/웹)와 백엔드 서버 간의 모든 통신은 반드시 HTTPS(TLS 1.3)로 이루어져야 합니다. 더 나아가 HSTS(HTTP Strict Transport Security)를 적용하여 브라우저나 앱이 HTTP로 통신을 시도하는 것 자체를 원천 차단해야 합니다.
* **Payload 단말간 암호화 (E2EE 적용 검토)**: HTTPS에만 의존하지 않고, 클라이언트단에서 처방전 이미지나 텍스트를 전송하기 직전에 서버의 '공개키(Public Key)'로 한 번 더 암호화하여 전송합니다. 오직 서버가 가진 '개인키(Private Key)'로만 복호화가 가능하므로, 중간 네트워크 장비에서 패킷을 뜯어보더라도 원본 처방전 이미지를 절대 볼 수 없습니다.
* **앱 인증서 고정 (Certificate Pinning)**: 모바일 앱(iOS/Android)을 개발할 때, 서버의 진짜 SSL 인증서 해시값을 앱 내부에 하드코딩(Pinning) 해둡니다. 해커가 가짜 인증서를 스마트폰에 심어서 중간에서 감청(Man-In-The-Middle)하려고 시도하면, 앱이 통신을 즉시 거부하도록 만듭니다.

## 2. 데이터베이스(Firestore) 및 저장소 암호화
서버 관리자의 계정이 탈취되거나 데이터베이스 자체가 뚫리더라도, 해커가 사용자의 복약 기록이나 처방전 정보를 읽을 수 없도록 방어해야 합니다.

* **필드 레벨 암호화 (Field-Level Encryption)**: Firestore가 기본적으로 제공하는 저장소 암호화(Data at rest encryption)에 의존하는 것을 넘어, 애플리케이션 레벨(FastAPI)에서 암호화를 수행해야 합니다. 사용자의 이름, 생년월일, 복용 중인 약물 이름 등을 DB에 넣기 직전에 AES-256 대칭키 알고리즘으로 암호화하여 저장합니다. 해커가 DB를 통째로 빼가더라도 알 수 없는 문자열만 보게 됩니다.
* **처방전 원본 이미지 보관함 분리 및 권한 제어**: 처방전 이미지는 일반적인 스토리지에 그대로 저장하면 위험합니다. AWS S3나 Firebase Storage를 사용하되 **서버 사이드 암호화(SSE)**를 활성화해야 합니다. 또한 이미지에 접근할 때 인증된 사용자 본인과 연결된 보호자에게만 매우 짧은 시간(예: 3~5분) 동안만 유효한 **서명된 임시 URL(Presigned URL)**을 발급하는 방식으로 접근을 철저히 통제해야 합니다.

## 3. 외부 AI API 전송 시 개인정보 비식별화 (Anonymization)
처방전을 분석하기 위해 OpenAI와 같은 외부 API로 데이터를 전송할 때 사용자의 민감 정보가 유출될 위험을 차단해야 합니다.

* **민감 정보 마스킹 (Masking)**: 클라이언트나 백엔드 서버에서 처방전 이미지를 OpenAI로 보내기 전에, '약품 정보와 무관한 개인 식별 정보'(환자 이름, 주민등록번호, 병원 이름 등) 영역을 검은색 박스로 가리는 전처리(Masking) 과정을 반드시 거쳐야 합니다.
* **OpenAI API 데이터 보호 약관 적용**: 무료 버전의 챗봇이 아닌, API(Zero Data Retention)를 사용할 경우 전송된 데이터가 AI 학습에 사용되지 않는다는 약관을 적용받는지 재확인하고 정책을 설정해야 합니다.

## 4. 의료 어플리케이션 특화 보안 (접근 제어 및 감사 로그)
* **보호자(Guardian) 접근 권한 철회 기능**: 현재 시스템은 보호자가 연결되면 환자의 기록을 볼 수 있습니다. 의료법과 개인정보보호 관점에서는, 환자(또는 정보 주체)가 언제든지 원할 때 보호자의 접근 권한을 '취소(Revoke)'할 수 있는 기능이 필수적으로 구현되어야 합니다.
* **감사 로그 (Audit Logging)**: 누가, 언제, 어디서(IP), 누구의 의료 데이터(처방전 기록 등)를 조회했는지 모든 액션을 별도의 안전한 로그 서버에 기록해야 합니다. 문제 발생 시 추적이 가능하도록 법적 근거 자료를 남기는 것입니다.
* **자동 로그아웃 및 생체 인증 결합**: 의료 앱은 화면을 켜둔 채 자리를 비웠을 때 타인이 볼 수 있는 위험이 있습니다. 앱 내에서 일정 시간 활동이 없으면 JWT 토큰을 즉시 만료시키고, 앱을 다시 열 때 FaceID나 지문 인식 등을 통해 본인 인증을 강제하도록 클라이언트 보안 로직을 추가해야 합니다.

## 5. 백엔드 아키텍처 및 인증 구조 고도화 (Clean Architecture & SOLID)
현재 코드는 초기 개발의 형태인 '단일 책임 원칙(SRP)' 위배 상태입니다. 향후 유지보수와 기능 확장을 위해 아래와 같은 아키텍처 개편을 강력히 권장합니다.

* **클린 아키텍처(Clean Architecture) 기반 계층 분리**:
  현재 하나의 라우터 파일에 섞여 있는 로직을 3개의 계층으로 완전히 분리하여 결합도를 낮춰야 합니다.
  1. **Controllers (Routers)**: HTTP 통신, 파라미터 수신 및 검증 (Pydantic) 수행
  2. **Services (Use Cases)**: 핵심 비즈니스 로직, 권한 체크, AI 연동 데이터 가공 담당
  3. **Repositories (Data Access)**: Firestore DB 쿼리 전담 (의존성 역전 원칙 적용)
* **의존성 주입(Dependency Injection) 적용**:
  FastAPI의 `Depends()` 기능을 적극 활용하여 데이터베이스 객체(`get_db`)나 외부 API 클라이언트를 주입받는 형태로 설계해야 합니다. 이를 통해 코드 간의 결합도를 줄이고 단위 테스트(Unit Test)가 가능한 환경을 만듭니다.
* **JWT 및 로그인 보안 체계 업그레이드**:
  현재 뼈대가 잡혀있는 JWT 인증 로직에서 패스워드 단방향 암호화 방식을 기존 `sha256_crypt`에서 보안 수준이 더 뛰어난 `bcrypt`로 상향 적용해야 합니다. 추가로, 짧은 만료 시간의 Access Token과 이를 갱신해주는 Refresh Token 메커니즘을 함께 도입하여 토큰 탈취에 대비한 안전망을 구축해야 합니다.
